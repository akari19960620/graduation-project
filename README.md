# ケアバンク

## サービス概要
**ケアバンク** は、介護施設を探すご家族が、 **簡単な診断で自分に合った施設を見つけられる** サービスです。

「何を基準に選べばいいかわからない」という悩みを、簡単な質問に答えるだけで解決し、自分の優先順位に合った施設を効率的に見つけることができます。



## アイデアが生まれたきっかけ
介護職として働いていた時、「住み慣れた家で過ごしたい」と考える高齢者の方が多かったですが、独居を心配されるご家族も多く話し合った末に施設の入居を検討したり、ご自宅でご家族の力を借りながら生活していたが介護度が高くなり介護負担が大きいためやむなく施設への入居を検討されたりと様々な理由から施設への入居を希望される方が多くいました。
施設はケアマネから数か所紹介してもらい、見学に行ったり施設の空き待ちをしている方が多かったように感じます。

ですが、施設といっても様々で特徴や特色も異なります。
事前の打ち合わせで細かく確認、すり合わせができるといいですが、実際に入居すると「見学時には気にならなかったけど、どうしても合わない。」等が出てきます。
100％マッチする施設を探すのは難しいかもしれませんが、「自分に合った施設」を簡単に見つけられ 相性のいい施設で穏やかに過ごせるようアプリで何かお手伝いができないかと考えたのがきっかけです。

## 課題の整理
**高齢者とそのご家族が施設に対する情報が少ない**

- 施設の種類を詳しく知らないため、ケアマネ頼りになっている。
- 取り合えす空き待ちして、入居してしまい後から不満が出てくる
- 施設同士を比較することができない
- 何を基準に選べばいいかわからない
- 自分に合った施設を探す手段が限られている
- 入居後に不満がでてくる

### 本当の課題

 **入居する高齢者本人の後悔やご家族の罪悪感**

## 想定ユーザー
**初めて施設探しをする高齢者のご家族**
- 年齢層は介護職時代に見てきた方々をイメージしたため40-50代で仕事や家事の合間に情報収集をすることを想定。
- ケアマネからの情報＋αで使用する

※高齢者は機械が苦手な方が多いため、メインはご家族。

## 環境設定
**隙間時間にスマートフォンやiPadで情報収集**
- 忙しい合間をぬってちょこちょこ情報収集ができるように、お気に入り登録ができる機能や閲覧記録を見れるようにする。

## 既存サービス・競合調査
**サービス名：ケアマップ**
- どんなことができるか：
現在地や地図から周辺施設を探せる。
住所、電話番号、ウェブサイトがすぐに確認できる。
お気に入り登録ができる。
※ケアマップには診断機能はない。

## 差別化を「体験価値」で表現する
初めて施設を探すご家族が、 **「何を基準に選べばいいかわからない」という悩みを、5〜10問の簡単な診断で解決** できる。

診断では、「設備が充実しているが料金が高い施設と、設備はシンプルだが料金が安い施設、どちらを選びますか?」などの トレードオフの質問に答えるだけで、自分が何を優先しているかが明確になり、納得できる施設選びができる。


## 価値を１文で表す
施設探し中の人が、簡単な診断で優先順位に合った施設を見つけられる


## このアプリで実現すること
### MVPで作る機能
　＜トップ画面、診断画面、診断結果画面、施設詳細画面、施設のリンク＞
診断はMVP時は５問程度のチェックボックス形式

### 本リリースで作る機能
　＜ユーザー登録・ログイン機能、マイページ、お気に入り登録機能、閲覧履歴保存機能、口コミの5段階評価と自由記述欄＞

5段階評価内容
 - スタッフの対応・雰囲気
 - 施設の清潔さ
 - 施設の設備・環境
 - 食事の質・味
 - 施設との連絡・情報共有
 
## このアプリの懸念点、対策
公開情報（施設名、住所、電話番号、サービス内容など）のみでは、施設の雰囲気や実際の様子がわからない

**MVP段階での対策**
- 診断機能に特化し、施設の基本情報のみを表示
- 施設の公式サイトへのリンクを表示し、詳細情報を確認できるようにする
- 「公開情報を元に作成しています」と明記
- 「実際の見学をおすすめします」と明記

**本リリース後の対策**
- 口コミ機能を追加し、実際に見学したユーザーの声を掲載


## 今後の展開・発展の方向性
- 口コミが充実することで実際の施設の雰囲気がよりイメージしやすくなる。
- 段階的に対応エリアを拡大していく
- 診断質問数をもう少し増やして、それぞれに合った施設の精度を高める。

## 使用予定の技術
**フレームワーク：Ruby on Rails**

**DB:PostgreSQL**

**デプロイ先：render**

**使用予定ライブラリ：MVPでは Rails標準機能のみで実装予定**

## 技術選定理由
**Ruby on Rails**

- MVPに必要な機能が標準機能で実装できる。
- 日本語情報が豊富で詰まったときに調べやすい。

**PostgreSQL**

- MVPのデータ構造を扱うのに十分で、カリキュラムでも学習しているため。

**render**

- 無料枠で始められるため
- PostgreSQLを使用できるため。
